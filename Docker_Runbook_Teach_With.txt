## Difference between the ENTRYPOINT and CMD command
CMD ["echo", "Hello"] in a DockerFile

''
FROM ubuntu:22.04
CMD ["echo", "Hello"]
''

- docker run <image_name> will print "Hello"
- docker run <image_name> echo "Hi" will print "Hi" instead of "Hello"

ENTRYPOINT ["ping"] in a DockerFile
''
FROM ubuntu:22.04
ENTRYPOINT ["ping"]
''

- docker run <image_name> google.com
- prints "ping google"

ENTRYPOINT + CMD Together
''
FROM ubuntu:22.04
ENTRYPOINT ["ping"]
CMD ["localhost"]
''

- docker run <image_name>
- prints "ping localhost"

''
FROM ubuntu:22.04

CMD ["tail", "-f", "/dev/null"]
''
- docker build -t copy-example .
- docker run copy-example cat /tmp/file.txt


ADD
''  
FROM ubuntu:22.04
ADD ./file.txt /tmp/file.txt
CMD ["tail", "-f", "/dev/null"]
''
- docker build -t add-example .
- docker run add-example cat /tmp/file.txt

## Example of persistent volumes

- # build the image
docker build -t my-nginx .

- # create me volume
docker volume create my-data

- # run container and mount the volume
docker run -d -p 80:80 -v my-data:/var/www/html my-nginx

- # run container with a name and mount the volume
docker run -d --name my-nginx-container -p 80:80 -v my-data:/var/www/html my-nginx

# confirm volume was created
docker volume ls

# confirm volume is mounted on the container
docker inspect my-nginx


When the container restarts, the volume data is safe and still there because the volume lives independently on the EC2 host — but by default Docker won't automatically restart the container if it stops or the EC2 reboots.

To handle automatic restarts, add --restart policy to your docker run:

docker run -d --name my-nginx-container -p 80:80 -v my-data:/var/www/html --restart unless-stopped my-nginx

--restart no               → never restart (default)
--restart always           → always restart, even after EC2 reboot
--restart unless-stopped   → restart always EXCEPT if you manually stopped it (recommended)
--restart on-failure       → only restart if container crashed with an error

You can prove this to your students with a quick demo:

# Step 1 - create and run container, add a file to the volume
docker run -d --name my-nginx-container -p 80:80 -v my-data:/var/www/html my-nginx
docker exec my-nginx-container bash -c "echo 'hello' > /var/www/html/test.txt"

# Step 2 - delete the container
docker rm -f my-nginx-container

# Step 3 - create a new container with same volume
docker run -d --name my-nginx-container-2 -p 80:80 -v my-data:/var/www/html my-nginx

# Step 4 - file is still there ✅
docker exec my-nginx-container-2 cat /var/www/html/test.txt
# output: hello

version: '3.8'
services:
  jira:
    image: atlassian/jira-software:10.0.0  # Pull the latest Jira Software image
    restart: unless-stopped # makes the stack recover after reboot.
    depends_on:
      - db
    ports:
      - "80:8080"
    volumes:
      - jira-data:/var/atlassian/jira
    environment:
      - ATL_JDBC_URL=jdbc:postgresql://db/jiradb
      - ATL_JDBC_USER=jirauser
      - ATL_JDBC_PASSWORD=jirapassword

  db:
    image: postgres:14
    restart: unless-stopped # makes the stack recover after reboot.
    environment:
      - POSTGRES_DB=jiradb
      - POSTGRES_USER=jirauser
      - POSTGRES_PASSWORD=jirapassword
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  jira-data:
  db-data:

In your docker-compose.yml, this line is a volume mapping.

db-data is the name of a Docker volume.
/var/lib/postgresql/data is the path inside the db container where PostgreSQL stores its database files.
So this means:

PostgreSQL writes its data into /var/lib/postgresql/data
Docker stores that data in a persistent named volume called db-data on your Docker Host

Why it matters?
Your database data survives container restarts.
If the db container is removed, the actual database files remain in db-data.
You can later reattach the same volume to a new db container and keep your data.

# simple approach to protect sensitive data

Step 1: Create a .env file
Create .env in the same folder as docker-compose.yml:

POSTGRES_DB=jiradb
POSTGRES_USER=jirauser
POSTGRES_PASSWORD=jirapassword
JIRA_DB_USER=jirauser
JIRA_DB_PASSWORD=jirapassword

Step 2: Update docker-compose.yml
Use variable substitution instead of hardcoded secrets:



services:
  jira:
    image: atlassian/jira-software:10.0.0
    restart: unless-stopped
    depends_on:
      - db
    ports:
      - "8080:8080"
    volumes:
      - jira-data:/var/atlassian/jira
    environment:
      - ATL_JDBC_URL=jdbc:postgresql://db/${POSTGRES_DB}
      - ATL_JDBC_USER=${JIRA_DB_USER}
      - ATL_JDBC_PASSWORD=${JIRA_DB_PASSWORD}

  db:
    image: postgres:14
    restart: unless-stopped
    environment:
      - POSTGRES_DB=${POSTGRES_DB}
      - POSTGRES_USER=${POSTGRES_USER}
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  jira-data:
  db-data:

Step 3: Run Docker Compose

cd /path/to/your/project
docker-compose config
docker-compose up -d

docker compose config checks the file and resolves the variables. Allowing Docker Compose to automatically read a .env file from the same folder as docker-compose.yml and substitute values into the compose file.

Why this is better
docker-compose.yml stays reusable and safe.
Secrets are kept in .env, which stays local.
You can change passwords without editing the compose file.
If you share the compose file, no secret values leak.
